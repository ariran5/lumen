## Подробности по задачам

### S1.1 — Top inset

**Проблема:** `LumenPageViewController.view` занимает всю площадь VC (под nav bar). Renderer цеплялся за `view.layer`, поэтому контент детальной HN-страницы наезжал на nav bar.

**Решение:** UIView-обёртка `contentView` поверх `view`. Constraints:
```swift
content.topAnchor.constraint(equalTo: host.safeAreaLayoutGuide.topAnchor)
content.bottomAnchor.constraint(equalTo: host.bottomAnchor)  // под home indicator уходит
content.leadingAnchor.constraint(equalTo: host.leadingAnchor)
content.trailingAnchor.constraint(equalTo: host.trailingAnchor)
```
Renderer теперь `Renderer(hostView: contentView)`. VirtualList в FastAppHost получает тот же контейнер.

Проверено визуально: detail page HN с аватаром и title — аватар сидит ниже nav bar, не наезжает.

### S1.2 — WebView smoke

Запустил `https://news.ycombinator.com` через `xcrun simctl launch -url ...` — WKWebView отрендерил полноценную HN страницу. AddressBar показал URL, home button есть. Никаких регрессий после refactor `BrowserView`.

### S1.3 — Promise.all

В прошлой сессии было записано как баг. Сейчас инструментировал fetch bridge (print на START / BG-COMPLETE / RESOLVE / RESOLVE-RETURNED) и переписал HN на Promise.all с трейсингом из JS.

**Результат:**
- 5 параллельных запросов — все 5 завершились корректно
- 20 параллельных запросов — все 20 завершились за ~0.8s
- URLSession.shared делает батчи по 6 (httpMaximumConnectionsPerHost), это нормально и не баг
- Промежуточный resolve каждого Promise срабатывает по мере поступления, финальный Promise.all возвращает массив в правильном порядке

**Что значит:** баг из session 001 не воспроизвёлся. Возможные причины:
- network/DNS флакс во время прошлой сессии
- `loadSequentially` имел собственный bug, маскированный под hang Promise.all
- какое-то промежуточное состояние fetch bridge до его финального fix'а

Все диагностические `print` убраны. `engine.onLog → print` оставлен в FastAppHost для DEBUG.

### S1.4 — Чистка

Удалены 4 файла + связанные кнопки StartPage и 3 @State флага. StartPage теперь показывает 2 примера (HN fast-app, HN web) — каждый кликается, заполняет адресную строку и коммитит.

### S1.5 — Storage

API:
```js
lumen.storage.get(key)        // string | undefined
lumen.storage.set(key, value) // void
lumen.storage.remove(key)     // void
lumen.storage.keys()          // string[]
lumen.storage.clear()         // void — удаляет все ключи с префиксом lumen.storage.*
```

Хранение в `UserDefaults.standard` под префиксом `lumen.storage.`. Self-test прошёл (set→get→remove→get→undefined). HN при тапе на новость сохраняет `visited.<id> = '1'`, при render row проверяет и затемняет (opacity 0.55).

⚠️ **Limit:** список после тапа не перерендеривается — `lumen.virtualList` ставит данные один раз. Visited становится видно при cell reuse во время скролла или при перезапуске. Это правильный кейс для M6 reconciler + reactive update — будет решено в Phase 2.

⚠️ **Per-origin модель** отложена на Phase 5. Сейчас storage глобальный.

## Открытые ограничения после S1

(Большинство — те же что были, но обновлены)

1. ~~Promise.all виснет~~ — закрыто, не воспроизводится.
2. **`lumen.render` + `lumen.virtualList` не сосуществуют** — `detach` одноразовый, при возврате на render-режим страница пуста. M6 (P2.1) даст clean решение.
3. **FlexLayout без intrinsic-sizing** — `auto` height у контейнера не вычисляется из детей. Обходим явным `height`.
4. **Reconciler (M6) ещё не реализован** — каждый `lumen.render` пересоздаёт всё CALayer-дерево; visited-status в HN list не обновляется на лету.
5. **Off-main computation (P2.2)** — всё ещё на main thread.
6. **WebView режим — статически проверен**. Регулярного теста нет.

## Команды для запуска

Без изменений с session 001:

```bash
xcodegen generate
xcodebuild -project Lumen.xcodeproj -scheme Lumen \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build
bun tools/dev-server.ts Examples/HN 8080
xcrun simctl launch "iPhone 17 Pro" com.lumen.browser -url http://localhost:8080
```

Дополнительно — для отладки JS console:
```bash
xcrun simctl launch --console-pty <DEVICE> com.lumen.browser -url http://localhost:8080
```
выведет в текущий терминал и `print(...)` из Swift, и `console.log(...)` из JS (через `engine.onLog`).
