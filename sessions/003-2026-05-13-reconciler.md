## TL;DR

Закрыт MVP **M6 Reconciler (Phase 2.1)**. Renderer теперь diff'ит prev/next дерево и переиспользует CALayer'ы между рендерами, а не сносит всё под ноль. Это:

- Убирает флэш при update (доказано unit-тестом на identity layer'а).
- Делает update в 1.5× быстрее initial mount на 1000 нод (3.0ms → 2.0ms).
- Открывает дорогу off-main (P2.2) — diff/apply отделимы.

Дополнительно: `lumen.virtualList(...)` теперь возвращает JS handle с `.reload()`. HN reader вызывает reload в `onPop` — после возврата с detail-страницы прочитанные новости сразу faded в видимых ячейках, без полной пересборки списка.

## Численные результаты

| Сценарий | Время |
|---|---|
| 1000 nodes — initial mount | 3.0 ms |
| 1000 nodes — update 5 props (opacity) | 2.0 ms |
| Unit-тесты (всё, 14 кейсов) | 0.04s total |

Update быстрее mount на ~33% — потому что не создаём CALayer'ы, не вставляем в дерево, не зовём `applyImage` (source не менялся), а только обновляем properties у существующих layer'ов.

Бюджет из ROADMAP — `< 2ms`. На симуляторе iPhone 17 Pro попали в 2.0ms. На железе ProMotion должно быть быстрее (cap 60Hz на симе делает CALayer-операции дороже). Подтвердим в M10 на физ. устройстве.

## Архитектурные решения и trade-offs

### Index-based vs Keyed LIS

Выбрал index-based как MVP. Почему:
- Наши текущие use-case'ы (HN list, HelloApp) — без перестановки детей. Diff на одном index'е работает идеально.
- LIS-move добавляет ~150 LOC и debug-сложности. Зарабатывает только когда есть массовая перестановка (drag-to-reorder, sort animations) — пока нет.
- `key` parsing уже есть в RenderNode — добавить LIS можно позже без breaking change.

**Когда добавить LIS:** появится feature вроде drag-and-drop списка, или animated reorder. Маркер — в коде HN/HelloApp видим явный `key:` на детях, а в demo идёт перестановка → значит время.

### MountedNode как class, не struct

Renderer должен мутировать `mounted.layer` при swap kind, `mounted.children` при add/remove. Со struct это значит передача `inout` или возврат нового value — болезненно при рекурсии. Class даёт identity + reference semantics.

Узкое место: цикл `tapHandlers: [ObjectIdentifier(layer): JSValue]` — ObjectIdentifier(layer) меняется при swap, поэтому в `reconcile` на kind change аккуратно вычищаем старый key до swap.

### Image не реdecode на update

`MountedNode.loadedImageSource` хранит последний загруженный URL. На reconcile if `newNode.source == loadedImageSource` — пропускаем `applyImage`. Это критично для HN: favicon'ы не должны мигать при reload списка.

### Render + virtualList coexistence — НЕ решено в этой сессии

Это остаётся ограничением: `lumen.virtualList` всё ещё side-channel API, который `detach`'ит renderer страницы и сам монтирует UICollectionView. Внутри ячеек reconciler работает, но смешать `lumen.render` и `lumen.virtualList` на одной странице нельзя.

Правильное решение — `kind: 'virtualList'` в самом дереве (см. P2.1.1 в ROADMAP). Это отдельная сессия.

## P2.1.1 — virtualList как kind (закрыто в той же сессии)

После закрытия M6 MVP сразу взяли P2.1.1, чтобы убрать архитектурный костыль side-channel `lumen.virtualList(config)` + `detach`.

### Что сделано

- `RenderNode.Kind.virtualList` + поля `listCount: Int`, `listItemHeight: CGFloat`, `listRenderFn: JSValue?`. Парсер видит `type: 'virtualList'` и считывает `count`/`itemHeight`/`render`.
- `MountedNode` обзавёлся `virtualListController` + `virtualListView`. На mount kind=virtualList Renderer создаёт overlay UICollectionView как subview `hostView` с absolute `flex.frame`; на reconcile — обновляет `count`/`itemHeight`/`renderFn` через `VirtualListController.update(...)`, тригерит `reloadData()`; на remove или kind swap — снимает overlay.
- `VirtualListController.count/itemHeight/renderFn` стали `var`. `view: weak VirtualListView?` хранит ссылку для reload.
- Удалён файл `JSEngine+VirtualList.swift` и его вызов в `FastAppHost`. Side-channel API больше нет — один путь через дерево.
- `lumen.virtualList(config)` как side-channel **снят полностью**, тоже handle с `.reload()` — снят. Все use-case'ы перешли на дерево.

### Обнаруженный баг и фикс в reconciler

При kind swap (`text` → `virtualList`) старый код в `reconcile` снимал old layer, ставил new layer, вызывал `applyAll`, а затем для **детей** свапнутого узла дёргал `mountFresh`. Сам swap'нутый узел через `mountFresh` не проходил — `mountVirtualList` не вызывался, UICollectionView не появлялась.

Фикс: после swap дополнительно проверяем `next.kind == .virtualList` и зовём `mountVirtualList` напрямую.

Также при swap чистим `mount.virtualListView` и `virtualListController`, чтобы старый overlay (если был) удалился перед монтированием нового.

### HN demo на новом API

`mountList()` теперь:

```js
lumen.render({
  type: 'view',
  style: { flex: 1, backgroundColor: '#0F0F12' },
  children: [
    { type: 'view', style: { flexDirection: 'row', height: 48, ... },
      children: [
        { type: 'text', text: '30 top stories', style: { flex: 1, ... } },
        { type: 'view', onTap: () => { lumen.storage.clear(); mountList() },
          style: { width: 110, height: 28, ... },
          children: [{ type: 'text', text: 'Clear visited', ... }],
        },
      ],
    },
    { type: 'virtualList', count: stories.length, itemHeight: 88, render: renderRow,
      style: { flex: 1 } },
  ],
})
```

`onPop` в `openStory` теперь вызывает `mountList()` — reconciler находит virtualList в дереве, обновляет controller и reload'ит видимые ячейки.

### Известное ограничение

`mountVirtualList` использует `flex.frame` напрямую как абсолютные координаты в hostView. Это правильно, когда virtualList сидит на верхнем уровне дерева (root flex-контейнер или его прямой ребёнок, как в HN). Для произвольной вложенности нужно `CALayer.convert(bounds, to: rootLayer)` — добавится при первом use-case с глубоко вложенным virtualList.

### Тесты

`Tests/ReconcilerTests.swift` обновлён:
- `testVirtualListKindParses` — поля `listCount`/`listItemHeight` доступны через struct
- `testVirtualListWithoutHostViewIsNoop` — Renderer с rootLayer-only не падает при kind=virtualList

Всего 9 ReconcilerTests + 7 FlexLayoutTests = 16/16 passing.

## P2.3 — Intrinsic-sizing (закрыто в той же сессии)

После P2.1.1 сразу взяли P2.3, чтобы убрать необходимость задавать `width: N` для контейнеров (видно было в HN header — `width: 110` для "Clear visited" вручную).

### Что сделано

- `FlexLayoutEngine.intrinsicSize(_:available:isParentRow:)` — рекурсивный helper. Для leaf'а возвращает `measure(available)` или `(0, 0)`. Для контейнера — bounding box детей + padding + gaps. Узлы с `flex > 0` в main axis считаются как 0 для intrinsic (shrink-to-fit, не grow-to-fill).
- `calculateLayout`:
  - в main-axis branch — fallback на `intrinsicSize` вместо `0` когда у ребёнка нет explicit/flex
  - в cross-axis branch — для контейнеров (`!children.isEmpty`) тоже считаем intrinsic, а не 0
- **Баг-фикс:** условие `cs.alignItems == .stretch` смотрело на child, но в CSS-семантике stretch определяется parent'ом. Убрал child-проверку, оставил только `node.style.alignItems == .stretch`. После этого intrinsic-sizing заработал по cross-axis тоже.

### Тесты (3 новых)

- `testIntrinsicRowFromChildren` — row container без explicit size, два ребёнка с явными размерами → root заполняет себя суммой детей + padding + gap.
- `testIntrinsicColumnFromChildren` — column со смешанными размерами по cross, гэп.
- `testIntrinsicWithTextMeasure` — wrapper c text leaf'ом → padding + measure result.

Все 3 passing. Общий счёт: **10 FlexLayout + 9 Reconciler = 19/19**.

### HN demo очищен

`mountList()` header теперь:
```js
{
  type: 'view',
  style: { flexDirection: 'row', alignItems: 'center', padding..., gap: 10, backgroundColor },
  children: [
    { type: 'text', text: '30 top stories', style: { flex: 1, fontSize, fontWeight, color } },
    { type: 'view', onTap: ..., style: { padding..., backgroundColor, borderRadius },
      children: [{ type: 'text', text: 'Clear visited', style: { fontSize, fontWeight, color } }],
    },
  ],
}
```

Без явных `width/height` для header, button, текстов. Layout сам считает.

### Ограничения

- Image-нод пока без intrinsic measure — пока картинка не загрузилась, размер неизвестен. Если у image нет explicit size — intrinsic = 0. Это допустимо: на image обычно ставят explicit `{width, height}`.
- `flex` ребёнок в main axis для intrinsic считается как 0 (shrink-to-fit). Это правильно — flex это grow, не intrinsic.
- Двойной measure для text leaf'ов на одном проходе: intrinsicSize вызывает measure, потом calculateLayout вызывает measure ещё раз. Возможна оптимизация через memo, но не критично сейчас.
