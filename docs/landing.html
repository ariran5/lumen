§ 01 — Тезис

JS — не узкое место.

Двадцать лет назад в браузер встроили модель документа. Сегодня этой моделью пользуются как платформой приложений — и за это платят кадрами, памятью и весом ввода. Заменяем не движок JS, а фундамент рендеринга.

«Тормоза браузера — это не про JS-движок. Главная боль — DOM, CSS и cascade.»
01

JS остаётся

JavaScriptCore — единственный JIT, разрешённый на iOS. Hermes без JIT медленнее в 3–5×. Менять не нужно — он не виноват.

02

HTML и CSS — нет

Минималистичная модель компонентов вместо DOM. Inline-стили вместо каскада. Каждый видимый узел = один CALayer.

03

Ниша пустая

Браузеров-оболочек с двумя движками — WebView плюс native fast-format — не существует. Ни на западе, ни как отдельный продукт нигде.

§ 02 — Архитектура

Один shell. Два движка.

Пользователь вводит URL. Lumen стучится за /.well-known/lumen.json. Есть манифест — поднимаем нативный runtime. Нет — отдаём управление WKWebView. Кэш TTL 24h, переключение бесшовное.

Engine A · Web

WKWebView

Совместимый рендер всего, что есть в открытом вебе. Без изменений и компромиссов.

  • Стандартный JS, DOM, CSS, history API
  • Полная совместимость с существующими сайтами
  • Прозрачный fallback, если нет манифеста
  • Edge-swipe back, навигация и история
or
Engine B · Native

LumenRuntime

JS-логика на JavaScriptCore, рендер прямо через CALayer. Native gestures, native animations, native scrolling.

  • Функциональные компоненты + signals
  • Reconciler с index-based diff (1k узлов → 2.0ms)
  • Flexbox на Swift, ~250 LOC
  • Off-main анимации через CABasicAnimation

LumenShell

Общая оболочка над обоими движками: address bar, табы, история, manifest discovery с per-host кэшем. Сам shell сейчас переписывается на Lumen — eat-your-own-dogfood.

§ 04 — DX

Просто JS. Без JSX, build-step, parser-а.

Flutter-style функциональные компоненты + signals в духе Vue Vapor / SolidJS. TypeScript-типы работают идеально, потому что нет ни одного транспилятора между вами и runtime.

Никаких build pipelines

Bun.Transpiler стрипает типы на лету. Edit → save → ~300ms → видишь результат на устройстве.

Fine-grained reactivity

Каждый prop в style — отдельный thunk. Меняется одно значение — обновляется один узел. Не дерево.

CLI: init / dev / build

@lumen/cli поднимает dev-server с WebSocket HMR, собирает bundle для production.

§ 05 — Правила

Три железных правила.

Когда выбираешь между «удобно» и «120 fps», правил три. Им подчиняется всё.

I.

Main thread рисует. Не считает.

Layout, диффинг, transform-матрицы — где угодно, кроме main. Main только коммитит готовый кадр. Даже если JS залип на 200ms, движение не дрогнет.

II.

Каждый видимый узел = один CALayer.

Никаких UIView (≈200 байт лишнего event-handling на штуку). Никаких лишних backing-store. Композит — забота GPU, иерархия — забота CoreAnimation.

III.

Bridge синхронный, но дешёвый.

Один вызов JS → Swift меньше 10μs. Через batch-патчи и стабильные ID-узлы. Async было бы корректнее, но медленнее на порядки.

Сознательно отброшено
DOM CSS cascade history API UIView Yoga (C++) JSX runtime parser

Каждый элемент списка — рассмотрен и явно отвергнут с письменным обоснованием в docs/IDEA.md. Yoga отвергнут уже после интеграции: 250 строк Swift покрыли все кейсы без C++/SPM трения.
