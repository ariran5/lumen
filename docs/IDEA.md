# Lumen — идея и обоснование

> Снимок состояния обсуждения на 2026-05-12. Этот документ описывает **что мы хотим сделать и почему**. План реализации — в [PLAN.md](PLAN.md). При сильных расхождениях между этим документом и тем, к чему мы пришли позже, нужно обновить и обсудить.

---

## 1. Что мы хотим построить

Альтернативный браузер с **двумя движками внутри одной оболочки**:

1. **WebView-движок** (системный WKWebView) — открывает обычный веб как Safari.
2. **Native Runtime** — открывает «быстрые сайты» в собственном формате, рендерящемся напрямую через CALayer/Metal, с JS-логикой на JavaScriptCore.

Браузер сам определяет, какой движок использовать, по манифесту `/.well-known/lumen.json`. Для пользователя — прозрачно. Для разработчика «быстрого сайта» — отдельный SDK и формат бандла.

Цель производительности: **120 fps на ProMotion, ощущение нативного приложения**, при сохранении возможности открывать любой обычный сайт через WebView.

---

## 2. Откуда взялась идея

Изначальный запрос: «браузер с нативной скоростью, в котором JS работал бы как нативный Swift».

В ходе обсуждения отделили два часто смешиваемых тезиса:

### JS — не узкое место

JavaScriptCore с JIT (Safari) уже близок к нативной скорости на compute-heavy задачах. «Тормоза браузера» — это не про JS-движок.

### Тормозит сама веб-модель

По убыванию вклада:

1. **Синхронный layout / forced reflow** — `offsetHeight`, `getBoundingClientRect` и подобное останавливают всё для пересчёта.
2. **CSS-каскад с descendant-инвалидацией** — изменение класса родителя приводит к перепроверке тысяч потомков. Селекторы матчатся справа налево.
3. **Retained DOM с наблюдаемыми мутациями** — DOM-узел это тяжёлый C++ объект с MutationObserver, accessibility-веткой, ranges.
4. **Один main thread на JS + layout + paint + ввод + GC**.
5. **Непредсказуемые границы композитора** — слой промоутится в GPU только при определённых условиях.
6. **Текст** — shaping, bidi, line-breaking при каждом layout.
7. **GC паузы** JS-кучи (5–15ms в major collection).
8. **Event loop семантика** — microtasks, rAF, синхронные скрипты.
9. **Accessibility tree** — параллельное дерево, синхронизируемое с DOM.
10. **Легаси-семантика** — quirks mode, `document.write`, `float`, `vertical-align`, синхронный `<script>`.

Вывод: «быстрый браузер для обычного веба» — это уровень Apple/Google, в одиночку не вытянуть. Реальная цель — **новая runtime-платформа**, где JS остаётся, а HTML/CSS заменены минималистичной моделью с native-mapping.

## 3. Прецеденты и почему ниши пустые

### Mini Programs в Китае (WeChat, Alipay, ByteDance)

Архитектурно ровно то, что мы хотим: WebView + альтернативный нативный рантайм в одной оболочке. **Но это супер-приложения**, не браузеры. Нет адресной строки, нет произвольного URL — только каталог внутри мессенджера. Используется миллиардом людей.

### Google AMP (2015–2022)

Самая близкая попытка на западе. Альтернативный HTML-формат, рендерился быстрее в Google-вьювере. Технически работало. **Умер не из-за технологии**, а из-за политики: Google форсил в выдаче, антимонопольщики наехали, издатели взбунтовались, в 2021 убрали приоритет, в 2022 заморожен.

**Урок:** альтернативный формат без независимой дистрибуции не выживает.

### Facebook Instant Articles, Apple News Format

Те же идеи, оба мертвы или нишевые.

### Hyperview, Sciter, Quick App

XML/HTML-подмножество рендерится нативно. Но это **SDK для приложений**, а не браузер.

### Lynx (ByteDance, опенсорс март 2025)

Готовый рантайм: JS + Yoga + native views, двухпоточная архитектура. Используется внутри TikTok. **Браузер вокруг него никто не построил.**

### Servo / Ladybird

Свои браузерные движки, но цель — стандартный веб, не альтернативный формат.

### Вывод

**Браузер-оболочки с двумя движками — WebView + native fast format — не существует ни на западе, ни как самостоятельный продукт нигде в мире.** Ниша пустая.

Почему так:
- **Дистрибуция:** на западе вход в интернет — Google Search и Safari/Chrome. Новый браузер без аудитории = новый формат без сайтов.
- **Сетевые эффекты против:** разработчики пишут под то, что у всех открывается.
- **Apple/Google не заинтересованы:** у них есть нативные App Store, второй формат им не нужен.

## 4. Реалистичные продуктовые пути

Технически делаемо, продуктово сложно. Четыре пути:

1. **Dev-tool, не массовый продукт.** Браузер для разработчиков, которым нужен быстрый UI: дашборды, internal tools, админки. Они сами и пользователи, и авторы — нет курицы-яйца. Может победить Electron.
2. **Вертикальная ниша.** Браузер для конкретной категории (чтение, торговля, gaming), где сам куратор контента. Контент-партнёры переписывают сайты под формат за быстрый UX.
3. **SDK + reference-браузер.** Открыть рантайм как Lynx. Браузер — демо, чтобы показать технологию. Зарабатывать на B2B-внедрениях в чужие приложения.
4. **Браузер для AI-агентов.** Альтернативный формат описан так, что LLM генерирует его лучше, чем HTML. Никем не занято, может стать актуально через 1–2 года.

**Текущее предпочтение не зафиксировано.** Решаем по итогам POC и первых пользовательских тестов.

## 5. Архитектурные решения (high-level)

### Топология

```
┌────────────────────────────────────────────────────────┐
│                  Lumen Shell (Swift)                   │
│   таб-бар, адрес, история, sync, settings, downloads   │
└──────────────┬─────────────────────────────────────────┘
               │ NavigationController
   ┌───────────┴────────────┐
   ▼                        ▼
WebTab (WKWebView)    FastTab (LumenRuntime)
                           │
   ┌───────────────────────┼────────────────────────┐
   ▼                       ▼                        ▼
 JS Engine            Reconciler              Renderer
 (JavaScriptCore)     (Swift)                 (CALayer + CoreText)
   │                       ▲                        ▲
   │ JSI-bridge            │ diff(prev, next)       │ apply patches
   └───→ host objects ─────┘                        │
                                                    │
                              Layout (Yoga / C++) ──┘
```

### Три железных правила

1. **Main thread рисует, не считает.** JS, layout, network, decode — всё уходит с main thread.
2. **Каждый видимый узел = один CALayer.** Никаких UIView-обёрток, кроме корня.
3. **Bridge синхронный, но дешёвый.** Никаких JSON-serialize-через-мост, как в старом RN.

### Технологический стек

| Слой | Выбор | Почему |
|---|---|---|
| Shell | Swift 6 + SwiftUI | Chrome не hot path, SwiftUI к месту |
| WebView | WKWebView, `WKProcessPool` на таб | Системный, ничего лучше нет |
| Persistence | SQLite через GRDB.swift | CoreData капризен, Realm — лишний рантайм |
| JS-движок | JavaScriptCore | JIT разрешён на iOS только ему. Hermes без JIT медленнее в 3–5x |
| Layout | Yoga (C++) | 12 лет в проде, тот же что в RN/Lynx |
| Renderer | CALayer напрямую, CoreText, Metal для эффектов | UIView добавляет ~200 байт и ненужный event-handling |
| Анимации | CAAnimation | Крутится в `backboardd`, free 120Hz даже при занятом main thread |
| Скролл | UIScrollView | Физика iOS вшита, заменять — потеря фила |
| Bridge | JSI-style direct C-API | `JSExport` слишком медленный (NSInvocation) |
| Bundler | esbuild на Bun | 10x быстрее Rollup, нативный TS/JSX |
| Bytecode | `JSScript.cacheBytecode()` | Парсинг JS = 30–100ms на старт, надо убрать |

### Bundle-формат

```
app.lumen  (gzip tar)
├── manifest.json
├── main.jsbc           # JSC bytecode
├── chunks/*.jsbc       # lazy chunks
└── assets/
    ├── icons/*.svg
    └── images/*.webp
```

### Манифест и discovery

`GET https://example.com/.well-known/lumen.json`

```json
{
  "name": "Example App",
  "version": "1.2.3",
  "bundle": "https://cdn.example.com/app-1.2.3.lumen",
  "bundle_hash": "sha256-...",
  "min_runtime": "0.1",
  "permissions": ["geolocation", "notifications"]
}
```

Flow: ввод URL → параллельно WKWebView грузит страницу + дёргается манифест → если есть → отменяем WebView, открываем fast mode.

### Что сознательно отброшено

| Решение | Что выбрали | Что отбросили |
|---|---|---|
| UI target | CALayer | SwiftUI (непредсказуемый перф), UIView (overhead) |
| Layout | Yoga | AutoLayout (медленно), своё (не успеем) |
| Стили | Inline props | CSS, каскад, селекторы |
| JS-фреймворк | Свой минимальный (~8KB) | React (45KB + reconciler-конфликт) |
| Шаблоны | JSX через esbuild | Свой DSL |
| Серверный рендер | Нет | Усложняет всё |
| Веб-стандарты | Подмножество (fetch, URL, crypto) | DOM, CSS, history API |

## 6. Производительные бюджеты

Это **целевые числа на M10**. Если не сходимся — Phase 0 не пройдена.

| Метрика | Цель |
|---|---|
| Холодный старт fast-таба до first paint (bundle кэширован) | <150ms |
| Без кэша, на 4G | <800ms |
| Кадр в скролле списка 10k items | <8ms (стабильные 120fps) |
| JS heap для типичного приложения | <20MB |
| RSS таба | <60MB (против ~150MB у WKWebView того же UX) |
| Bridge call | <10μs (sync) |

## 7. Главные риски

1. **App Store policy.** Apple может счесть, что bundle с JS = «альтернативный магазин приложений». Контрмера: bundle — это данные, не код, аналогия с веб-страницей. Готовиться к юридическому разговору.
2. **JSC bytecode-формат закрытый.** API стабильный, но формат меняется между версиями iOS. Решение: bytecode генерируется на устройстве при первой загрузке, кэшируется.
3. **JIT отключается в Low Power Mode** и для background-табов. Не критично, но измерять.
4. **Cookie sharing между WebView и Fast Mode** — `WKHTTPCookieStore` vs URLSession. Решаемо через shared `HTTPCookieStorage`, но грабли.
5. **Курица-яйцо с контентом.** Технически решаемо, продуктово — главная проблема. См. раздел 4.

---

## 8. Что значит «не получилось»

POC (Phase 0) считается провальным, если на M10:

- Скролл 10k items не выдаёт стабильные 110+ fps на ProMotion устройстве,
- ИЛИ cold start превышает 200ms на кэшированном бандле,
- ИЛИ Memory RSS превышает 80MB.

В этом случае — стоп, разбор причин, переосмысление архитектуры. Возможно, гипотеза «native rendering из JS даёт ощутимый прирост над WKWebView» окажется неверной, и тогда проект меняется или закрывается.
